Configuration by Exception

JPA's core design principle: sensible defaults mean you only configure what's different.

DEFAULT (ZERO CONFIGURATION)

With just @Entity and @Id, JPA automatically maps:

  • Class name → Table name
  • Field names → Column names
  • Java types → SQL types (JDBC rules)
  • All fields → Persistent columns
→
EXCEPTION (WHEN YOU CUSTOMISE)

Add annotations only to override defaults:

  • @Table → Change table name
  • @Column → Change column properties
  • @GeneratedValue → Auto-generate IDs
  • @Transient → Exclude a field
Key insight: The less you annotate, the more you rely on defaults. Annotation is the exception, not the rule.

Customising Tables & Columns

Overriding JPA's default mappings with @Table, @Column, and @SecondaryTable annotations.

@Table — Rename the Table

Use @Table(name="...") when you need the table name to differ from the class name.

DEFAULT
@Entity
public class Book { ... }

↓ Maps to table BOOK

→
CUSTOMISED
@Entity
@Table(name = "t_Book")
public class Book { ... }

↓ Maps to table t_Book

The @Table annotation can also set catalog and schema for more complex database structures.

@Column — The Customisation Powerhouse

Fine-tune individual column properties. Every parameter has a sensible default.

ParameterDefaultPurpose
nameField nameRename the column
nullabletrueAllow NULL values
length255Max string length
uniquefalseEnforce uniqueness
updatabletrueAllow value changes after insert
insertabletrueInclude in INSERT statements
updatable = false means you can change the value in Java memory, but it won't sync to the database. The column is "locked" after the first INSERT.

@Column — Before & After

Customised Book Entity

@Entity
@Table(name = "MyFavouriteBook")
public class Book {
  @Id @GeneratedValue
  private Long id;

  @Column(name = "Book_Title",
         nullable = false,
         updatable = false)
  private String title;

  private Float price;

  @Column(length = 2000)
  private String description;

  private String isbn;

  @Column(name = "Number_of_Pages",
         nullable = false)
  private Integer nbOfPage;

  private Boolean illustrations;
}

Resulting Database Table

MyFavouriteBook

Book_Title VARCHAR(255) NOT NULL
PRICE FLOAT
DESCRIPTION VARCHAR(2000)
ISBN VARCHAR(255)
Number_of_Pages INT NOT NULL
ILLUSTRATIONS SMALLINT
What changed: Table renamed, two columns renamed, two columns made NOT NULL, description extended to 2000 chars, title locked from updates.

Primary Keys & @GeneratedValue

Every entity needs @Id. The value can be assigned manually or auto-generated.

Manual Assignment

@Id
private Long id;
// Application must set id before persist

Auto-Generated

@Id
@GeneratedValue
private Long id;
// Provider assigns id automatically

Generation Strategies

StrategyHow It Works
AUTOProvider picks best strategy (default)
IDENTITYDatabase auto-increment column
SEQUENCEDatabase sequence object
TABLESeparate table stores next ID values
Rule: Once assigned, a primary key value can never be updated.

@Basic — Fetch Strategies

Controls whether an attribute is loaded immediately or on-demand.

EAGER (default)

Load the attribute immediately when the entity is fetched.

Use for: Small fields you always need — names, prices, flags.

@Basic(fetch = FetchType.EAGER)
private String title;

LAZY

Load the attribute only when the application accesses it.

Use for: Large fields — BLOBs, file data, long text.

@Basic(fetch = FetchType.LAZY)
private byte[] wavFile;
Performance tip: A music Track entity storing a 10MB WAV file should use LAZY fetch — you don't want to load that binary every time you list tracks.

More Attribute Annotations OPTIONAL

@Transient

Exclude a field from persistence. Useful for computed values.

@Transient
private Integer age;
// Calculated from dateOfBirth

@Temporal

Map java.util.Date to DATE, TIME, or TIMESTAMP.

@Temporal(TemporalType.DATE)
private Date dateOfBirth;

@Enumerated

Store enums as ORDINAL (int) or STRING (text). Prefer STRING.

@Enumerated(EnumType.STRING)
private CreditCardType type;
// Stores "VISA", not 0
Why prefer STRING over ORDINAL? If you reorder or insert new enum values, ordinal numbers shift — existing database data becomes incorrect. String storage is safe against reordering.

Quick Check — ORM Customisation

You annotate a field with @Column(updatable = false). What happens when you call setTitle("New Title") on a managed entity?

A. A runtime exception is thrown.
B. The setter is blocked — the value doesn't change in memory.
C. The value changes in memory but is NOT written back to the database.
D. The value changes both in memory and in the database.
Correct! updatable = false only prevents the database UPDATE — Java setters still work normally. The in-memory value changes, but the persistence provider won't include that column in UPDATE statements.
Not quite. updatable = false doesn't affect Java code — the setter works fine and the value changes in memory. But the persistence provider simply won't include that column in UPDATE SQL statements, so the change never reaches the database.

@OneToOne Unidirectional — Concept

A customer has exactly one address. We only navigate Customer → Address.

Customer Entity

@Entity
public class Customer {
  @Id @GeneratedValue
  private Long id;
  private String name;

  // Entity as attribute = relationship
  private Address address;
}
No @OneToOne needed! JPA infers a one-to-one relationship when an entity attribute references another entity.

Address Entity

@Entity
public class Address {
  @Id @GeneratedValue
  private Long id;
  private String street;
  private String city;
  private String country;

  // No reference back to Customer
  // → Unidirectional
}

Default Database Result

CUSTOMER table gets an ADDRESS_ID foreign key column pointing to ADDRESS table.

@OneToOne — Customised Mapping

Use @OneToOne + @JoinColumn to control the relationship and foreign key.

@Entity
public class Customer {
  @Id @GeneratedValue
  private Long id;
  private String name;

  @OneToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "ADD_FK",
             nullable = false)
  private Address address;
}

@OneToMany Unidirectional — Concept

One order contains many order lines. Navigate only from Order → OrderLines.

Order Entity

@Entity
public class AnOrder {
  @Id @GeneratedValue
  private Long id;

  @Temporal(TemporalType.TIMESTAMP)
  private Date creationDate;

  // Collection of entities = OneToMany
  private List<OrderLine> orderLines;
}

OrderLine Entity

@Entity
public class OrderLine {
  @Id @GeneratedValue
  private Long id;

  private String item;
  private Double unitPrice;
  private Integer quantity;

  // No reference back to Order
  // → Unidirectional
}
Default behaviour: JPA creates a join table named ANORDER_ORDERLINE with two foreign key columns. No annotation needed for this default.

@OneToMany — Customised Mapping

You can rename the join table, change FK column names, or switch to a foreign key strategy entirely.

Option A: Custom Join Table

@OneToMany
@JoinTable(
  name = "jnd_ord_line",
  joinColumns =
    @JoinColumn(name = "order_fk"),
  inverseJoinColumns =
    @JoinColumn(name = "line_fk")
)
private List<OrderLine> orderLines;

Still uses a join table, but with custom names.

Option B: Foreign Key (No Join Table)

@OneToMany(fetch = FetchType.EAGER)
@JoinColumn(name = "ORDER_FK")
private List<OrderLine> orderLines;

Puts the FK directly in the ORDERLINE table. No intermediate table needed.

@SecondaryTable — One Entity, Multiple Tables OPTIONAL

Spread an entity's attributes across multiple tables — useful for legacy databases or very wide tables.

@Entity
@SecondaryTables({
  @SecondaryTable(name = "CITY"),
  @SecondaryTable(name = "COUNTRY")
})
public class Address {
  @Id private Long id;
  private String street;

  @Column(table = "CITY")
  private String city;

  @Column(table = "COUNTRY")
  private String country;
}

Composite Primary Keys OPTIONAL

When a single column isn't enough — combine multiple fields into one primary key.

1. Create a PK Class

@Embeddable
public class NewsId
       implements Serializable {
  private String title;
  private String language;

  // Must override equals() & hashCode()
  // Must have no-arg constructor
}

2. Embed in Entity

@Entity
public class News {
  @EmbeddedId
  private NewsId id;
  private String content;
}

Why Use Composite Keys?

Legacy databases — existing schema uses multi-column PKs

Business rules — e.g., news identified by title + language code

Finding by Composite Key

NewsId id = new NewsId("JPA Tips", "EN");
News news = em.find(News.class, id);
Best practice: Prefer a single auto-generated PK when possible. Use composites only when required by business logic or legacy constraints.

@Embeddable — Reusable Value Objects OPTIONAL

Embeddables have no identity of their own — their fields merge into the owning entity's table.

@Embeddable
public class Address {
  private String street;
  private String city;
  private String country;
  // No @Id — not an entity!
}

@Entity
public class Customer {
  @Id @GeneratedValue
  private Long id;
  private String name;

  @Embedded
  private Address address;
}

Result: Single Table

CUSTOMER
ID (PK)
NAME
STREET ← from Address
CITY ← from Address
COUNTRY ← from Address
Entity vs Embeddable: Entities have their own table and identity (@Id). Embeddables are "absorbed" into the parent entity's table — no separate table, no ID.

Quick Check — Relationships

An Order entity has a List<OrderLine> field with no JPA annotations. What does the persistence provider create by default?

A. A foreign key column in the ORDER table.
B. A join table named ORDER_ORDERLINE with two foreign key columns.
C. A foreign key column in the ORDERLINE table.
D. No relationship — you must add @OneToMany explicitly.
Correct! For a unidirectional one-to-many relationship, JPA's default strategy is a join table. The table name follows the convention: OWNING_TARGET (e.g., ORDER_ORDERLINE).
Not quite. JPA infers a @OneToMany relationship from the Collection type and defaults to a join table strategy. To use a foreign key instead, you'd need to explicitly add @JoinColumn.

Week 2 Summary

ORM Customisation

@Table renames the table. @Column controls column name, length, nullability, updatability. @GeneratedValue auto-generates primary keys. @SecondaryTable splits across tables.

Attribute Types

@Basic controls fetch strategy (LAZY/EAGER). @Transient excludes fields. @Temporal maps dates. @Enumerated stores enums as ordinal or string.

Relationships

Direction: unidirectional or bidirectional. Cardinality: @OneToOne, @OneToMany, @ManyToOne, @ManyToMany. All can be inferred from field types with zero annotations.

Join Strategies

Join Column (FK in owning table) — best for 1:1 and M:1. Join Table (intermediate table) — default for 1:N unidirectional. Use @JoinColumn or @JoinTable to switch or customise.
