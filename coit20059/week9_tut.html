Before you startPrerequisites & setup

  • A Jakarta EE / JF web project that runs on your application server (extend your Week 8 work if required).
  • Facelets as the view handler (javax.faces.DEFAULT_SUFFIX / .xhtml pages).
  • A managed bean in session or view scope for form state you need across requests.
Tip
Confirm faces-config.xml (or annotations) register your beans and that the server log shows no deployment errors before editing views.

Task AFacelets template & composition

Goal: Avoid duplicating chrome (header, footer, navigation) on every page.

  1. Create a template page (e.g. /WEB-INF/template.xhtml) with ui:insert regions for title, body, and optional sidebar.
  2. Create a client page using ui:composition template="..." and ui:define to fill those regions.
  3. Optional: extract repeated fragments with ui:include or a composite component if your tutor expects reuse.
Check
Two different URLs should render inside the same chrome—only the inner content changes.

Task BForms & EL bindings

Build a simple flow with h:form, inputs, and action:

<h:form>
  <h:inputText value="#{bean.name}" />
  <h:commandButton value="Submit" action="#{bean.submit}" />
</h:form>
  • Use EL (#{}) for two-way binding on inputs.
  • Return a navigation outcome string or use implicit navigation as taught in Week 8–9 lectures.
  • Add h:messages or field-level h:message after you attach validators.

Task C (stretch)Ajax & components

Enhance interactivity without full page reloads:

  • Wrap part of the tree in f:ajax or use primefaces/omnifaces only if your unit permits—otherwise stay with core JSF.
  • Re-render a small region (e.g. a panel listing rows) when a button fires.
Note
Match component IDs and render targets carefully; duplicate IDs in included fragments are a common source of errors.
